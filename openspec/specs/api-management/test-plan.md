## 5. 发现的 Bug

### BUG-API-001: 删除文档路由不匹配 [P1]

**描述**: 后端路由定义为 `DELETE /api-documents/:id`，但项目约定其他路由均使用 `POST /:id/delete` 模式

**影响**: 删除文档功能无法正常工作

**复现步骤**:
1. 在前端点击删除文档
2. 观察 Network 请求

**实际结果**: 前端发送 POST 请求到 `/api-documents/${id}/delete`，后端期望 DELETE 方法
**预期结果**: 路由一致

**根因**: 后端路由未遵循项目约定（其他路由均使用 `.post("/delete", ...)`）

**同类排查**:
- api-endpoints.ts: `.post("/delete", ...)` ✅
- test-cases.ts: `.post("/delete", ...)` ✅
- projects.ts: `.post("/delete", ...)` ✅
- environments.ts: `.post("/delete", ...)` ✅
- test-chains.ts: `.post("/delete", ...)` ✅
- test-plans.ts: `.post("/delete", ...)` ✅
- **api-documents.ts: `.delete("/:id", ...)`** ❌ 唯一不一致

**修复建议**: 
- 后端改为 `.post("/:id/delete", ...)` 以保持一致性

**Bug 归类**: 契约不一致

**状态**: 待修复

---

### BUG-API-002: confirm-update 缺少 content 字段 [P1]

**描述**: 后端 `confirm-update` 接口要求 `content` 字段，但技术方案设计文档中未定义该字段

**影响**: API 更新确认功能无法正常工作

**复现步骤**:
1. 导入 API 文档
2. 修改文档内容后重新上传
3. 点击「确认更新」

**实际结果**: 后端返回 400 错误 "content 必填"
**预期结果**: 确认更新成功（仅需 contentHash 和 accept 列表）

**根因分析**:
- 技术方案设计：`confirmUpdate` 仅需 `contentHash` + `acceptAdded/Modified/Removed`
- 后端实现：重新解析 `content` 获取端点列表
- 差异原因：后端未存储 diff 结果，导致需要重新传 content

**修复建议**: 
- 方案 A（推荐）：前端在 confirm-update 时传递原始 content（需存储在 import-api-doc-sheet 中）
- 方案 B：后端在 import 时存储 diff 结果，confirm-update 时从存储读取

**Bug 归类**: 契约不一致

**状态**: 待修复
